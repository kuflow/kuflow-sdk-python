# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`kuflow-sdk-python` is a **Poetry monorepo** containing all of KuFlow's published Python packages. Each top-level `kuflow-*` directory is an independently versioned, independently published package, but all packages **share a single version number** (PEP 440) driven by `scripts/create_local_version.sh`. Local cross-package dependencies use `{ path = "...", develop = true }` so changes propagate without re-installing.

## Packages & Dependency Graph

Listed in topological order (a package's dependencies come before it). This order is canonical and lives in `scripts/projects.sh` — keep it in sync when adding a module.

- **`kuflow-rest`** — REST client for the KuFlow API. Foundation package; only depends on `azure-core` + `isodate`. Most other packages depend on it.
- **`kuflow-robotframework`** — Robot Framework keyword library (depends on `kuflow-rest`).
- **`kuflow-temporal-common`** — Shared Temporal glue: KuFlow ↔ Temporal data converters, exceptions, auto-heartbeater (depends on `kuflow-rest` + `temporalio`).
- **`kuflow-temporal-activity-kuflow`** — Temporal Activities backed by the KuFlow API (depends on `kuflow-temporal-common`).
- **`kuflow-temporal-activity-robotframework`** — Temporal Activities that run Robot Framework robots (depends on `kuflow-temporal-common`).
- **`kuflow-temporal-workflow-kuflow`** — Workflow-side helpers (attributes, uuid7) safe to use inside Temporal workflow sandboxes (depends on `kuflow-temporal-common`).
- **`kuflow-temporal-worker`** — Temporal worker setup: connection config, authentication, and end-to-end payload **encryption** (codec/converter/interceptor) (depends on `kuflow-temporal-common` + `cryptography`).
- **`kuflow-temporal-sample`** — Example usage. Private, not published.

## Key Architectural Pattern: Generated + Patch in `kuflow-rest`

`kuflow-rest` is built around **AutoRest code generation** from the KuFlow OpenAPI spec. Understanding this layering is essential before editing it:

- **`kuflow_rest/_generated/`** — Machine-generated by AutoRest. **Do not hand-edit.** It is excluded from ruff linting (see `[tool.ruff.lint.per-file-ignores]` in root `pyproject.toml`) and will be overwritten on the next codegen run.
- **`kuflow_rest/` (top level)** — Hand-written public API that wraps the generated layer:
  - `_kuflow_rest_client.py` — `KuFlowRestClient`, the public entry point; wraps the generated client and exposes hand-written operation classes.
  - `operations/` — Thin, ergonomic wrappers over generated operations (e.g. `_process_operations.py`). Each takes the generated client and re-exposes methods with friendlier signatures.
  - `models/_models.py`, `utils/_parser.py` — Public models and helpers.
- The `_patch.py` convention (an AutoRest mechanism) lets you override generated behavior without editing generated files.

### Regenerating the REST client

Codegen lives in `kuflow-rest/openapi/`. It is a Node toolchain (AutoRest) driven by **pnpm** (mirroring the Java SDK; pnpm is pinned in `mise.toml` and the lockfile is `pnpm-lock.yaml`), configured by `openapi/readme.md`, which pins the exact OpenAPI spec commit via `input-file`.

```bash
cd kuflow-rest/openapi
pnpm install
pnpm run generate    # runs check-python → autorest 3.10.9 → ruff format → ruff check --fix → fix-discriminators.js
```

The `generate:workaround` step (`scripts/fix-discriminators.js`) post-processes generated discriminator handling. Bumping the API version = changing the `input-file` URL/commit in `openapi/readme.md`, then regenerating.

## Temporal Integration Concepts

- **Data conversion** (`kuflow-temporal-common/_converter.py`): KuFlow REST models are `azure-core` `Model` objects. `register_serializable_models()` plus `KuFlowModelJSONEncoder`/`KuFlowModelJSONTypeConverter` teach Temporal's payload converter how to (de)serialize these models across the workflow/activity boundary.
- **Encryption** (`kuflow-temporal-worker/_encryption/`): Optional end-to-end payload encryption via a custom Temporal payload codec + interceptor, with a KMS-backed key cache. The worker authenticates to KuFlow and can notify worker information.
- **Workflow sandbox safety**: code intended to run *inside* a Temporal workflow (deterministic sandbox) lives in `kuflow-temporal-workflow-kuflow`; code that performs I/O lives in the activity/worker packages.

## Development

Per `poetry.toml`, virtualenvs are **in-project** (`.venv/`), and **each package has its own `.venv`** — so an IDE/`poetry run` inside a package folder uses that package's pinned tool versions. The root package also has a `.venv` aggregating everything via the develop-mode path deps.

Targets: Python `>=3.9,<4.0`. Requires Poetry `>=2.0,<3.0` (repo pins `python 3.13.2` / `poetry 2.2.1` in `.tool-versions`).

```bash
# Set up the whole monorepo (from repo root)
poetry install

# Run a command in EVERY package, in topological order
./scripts/poetry_run_on_each.sh <cmd>      # e.g. update, lock, install, build, publish
                                            # `publish` auto-skips packages marked "Private :: Do not Upload"
```

### Lint & Type-check

Ruff is configured at the **root** `pyproject.toml` (line length 120; rule sets E/W/F/I/C/B/UP). The entire `_generated` tree is excluded.

```bash
ruff format          # format
ruff check --fix     # lint + autofix
poetry run mypy <path>
```

### Tests

Tests use PyTest; files are named `test_*.py` and live in each package's `test/` directory. Run them inside the relevant package so the correct `.venv` is used:

```bash
cd kuflow-rest && poetry run pytest                          # one package
cd kuflow-rest && poetry run pytest test/test_unit.py        # one file
cd kuflow-rest && poetry run pytest test/test_unit.py::test_name   # one test

./scripts/poetry_run_on_each.sh run pytest                   # all packages
```

## Versioning & Release

All packages share one version. **Do not** bump versions by hand across files — use the script, which rewrites every `pyproject.toml`, every `VERSION` file, and the `__version__`/`VERSION` constants in package `__init__.py`/`_version.py` files.

```bash
# 1. Edit the VERSION variable inside scripts/create_local_version.sh (PEP 440; dev versions end in .devN)
./scripts/create_local_version.sh
# 2. Re-lock everywhere
./scripts/poetry_run_on_each.sh lock && poetry lock
```

When **adding a new module**: add it to `scripts/projects.sh` (topological order) and add a matching `sed` line for its `__version__` in `scripts/create_local_version.sh`.

For CI builds, `scripts/prepare_monorepo_build.sh` rewrites the local `path =` dependencies into pinned version constraints (`^<version>`) before building wheels.
